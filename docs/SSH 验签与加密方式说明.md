# SSH 验签与加密方式说明（面试可用版）

## 1. 什么叫“可通过验签”？

一句话定义：

> 给定同一条消息 `m`、签名 `sig`、公钥 `pk`，执行 `Verify(pk, m, sig)` 返回 `true`，就叫“验签通过”。

数学/流程上等价于：

1. 客户端用私钥 `sk` 生成签名：`sig = Sign(sk, m)`
2. 服务端拿公钥 `pk` 验证：`Verify(pk, m, sig)`
3. 返回 `true` 则通过，`false` 则失败

在 GitHub SSH 里，你可以理解成：

- GitHub 发一个一次性挑战数据（challenge）
- 你的本机用私钥对 challenge 签名
- GitHub 用它保存的公钥验签
- 验签通过就说明“你确实持有对应私钥”

---

## 2. 验签失败常见原因

只要下面任何一条出现，就会失败：

1. **公钥不匹配**  
   GitHub 保存的公钥不是你这把私钥对应的公钥。
2. **消息被改动**  
   只要 `m` 变了，原签名通常就无效。
3. **签名被破坏/截断**  
   `sig` 内容损坏、格式错、编码错。
4. **算法不一致**  
   例如用 `ed25519` 的签名，按别的算法去验。
5. **没有把公钥登记到正确位置**  
   比如你把 key 加到另一个 GitHub 账号。

## 3. “私钥签名 + 公钥验签”和“公钥加密 + 私钥解密”的区别

这两者都属于非对称密码学，但目标不同。

### 3.1 私钥签名 + 公钥验签（数字签名）

- 目的：证明身份、保证完整性、防篡改
- 密钥方向：私钥 -> 签名，公钥 -> 验签
- 解决的问题：
  - “是不是你发的”
  - “内容是否被改过”
- 不直接提供“保密性”（别人仍可能看到明文）

典型场景：

- SSH 身份认证
- 软件包签名
- JWT/证书签名校验

### 3.2 公钥加密 + 私钥解密（非对称加密）

- 目的：保密（机密性）
- 密钥方向：公钥 -> 加密，私钥 -> 解密
- 解决的问题：
  - “除了私钥持有者，别人都看不懂内容”
- 不能直接证明“是谁发的”

典型场景：

- 把敏感数据安全发送给特定接收者

### 3.3 一张表看懂

| 机制 | 主要目标 | 谁做第一步 | 谁做第二步 | 是否提供保密性 | 是否证明身份 |
|---|---|---|---|---|---|
| 私钥签名 + 公钥验签 | 身份与完整性 | 私钥持有者签名 | 任意人用公钥验签 | 否 | 是 |
| 公钥加密 + 私钥解密 | 保密 | 任何人可用公钥加密 | 私钥持有者解密 | 是 | 否（单独看） |

## 4. 对称加密是什么？

对称加密 = 加密和解密使用**同一把密钥**。

例如：

- AES-GCM
- ChaCha20-Poly1305

特点：

1. **非常快**，适合大量数据传输
2. **密钥分发困难**：双方必须先安全共享同一把密钥
3. 常配合非对称机制使用（先协商会话密钥，再用对称加密传大数据）

---

## 5. 三者关系：真实系统怎么组合用？

在 SSH / TLS 这类协议中，通常是“混合方案”：

1. 用非对称机制做身份认证/密钥交换（含签名验证）
2. 协商出一次性会话密钥（session key）
3. 后续大流量数据用对称加密传输

为什么这样设计：

- 非对称：安全但慢
- 对称：快但需要先有共享密钥
- 组合后：兼顾安全和性能

---

## 6. 回到 GitHub SSH：为什么一定要上传公钥？

因为 GitHub 只有拿到你的公钥，才能做这一步：

`Verify(你的公钥, challenge, 你的签名)`

如果没有你的公钥，GitHub 无法验证你签名的合法性，就会拒绝连接并报：

```text
Permission denied (publickey)
```

## 7. 面试可直接回答的 20 秒版本

> SSH 登录用的是非对称密码学里的数字签名。客户端用私钥对 challenge 签名，服务端用预先登记的公钥验签。验签通过说明客户端持有对应私钥。签名机制主要解决身份认证和完整性；而公钥加密/私钥解密主要解决保密。实际系统常用混合方案：非对称做认证和密钥交换，对称做数据传输。
